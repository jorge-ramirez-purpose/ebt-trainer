# Step 1 Study Notes — Backend Project Structure

## What we built

A Node.js + Express REST API that serves the 310 German citizenship test questions from a PostgreSQL database. The React frontend now fetches from this API instead of importing a static file, with a fallback to the static data if the server is down.

---

## Key concepts

### Why a separate `server/` folder?

The backend is a completely different runtime environment from the frontend:

| | Frontend | Backend |
|---|---|---|
| Runtime | Browser | Node.js |
| Module system | ESM (`"type": "module"`) | CommonJS (`require`) |
| Build tool | Vite | `tsc` + `ts-node-dev` |
| Package config | root `package.json` | `server/package.json` |

Keeping them in separate folders lets each have its own `package.json`, `tsconfig.json`, and dependencies without conflicting.

---

### Why `"type": "commonjs"` in `server/package.json`?

Node.js has two module systems:

- **ESM** (`import`/`export`) — what the browser and modern bundlers use
- **CommonJS** (`require`/`module.exports`) — the original Node.js system

Most Node.js server libraries (Express, Prisma) are designed for CommonJS. Setting `"type": "commonjs"` in `server/package.json` tells Node.js to treat `.js` files as CommonJS, even though TypeScript source uses `import`/`export` syntax (TypeScript compiles those to `require` calls).

---

### What is Prisma?

Prisma is an **ORM** (Object-Relational Mapper) — a tool that lets you interact with a database using TypeScript instead of writing raw SQL.

**Without Prisma (raw SQL):**
```ts
const rows = await db.query("SELECT * FROM questions WHERE id = $1", [1]);
```

**With Prisma:**
```ts
const row = await prisma.question.findUnique({ where: { id: 1 } });
```

Prisma has three parts:

| Part | File | Purpose |
|---|---|---|
| Schema | `prisma/schema.prisma` | Defines the database tables as TypeScript-like models |
| Client | generated by `prisma generate` | The typed TS client you import in code |
| Migrations | created by `prisma migrate dev` | SQL files that create/alter the actual tables |

### The `Question` schema design

```prisma
model Question {
  id        Int      @id          // NOT auto-increment — questions have stable IDs 1–310
  question  String
  optionA   String
  optionB   String
  optionC   String
  optionD   String
  answer    Int                   // 0, 1, 2, or 3 (the index of the correct option)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

**Why four separate columns instead of a JSON array?**

The frontend type uses `options: [string, string, string, string]` — an array. But relational databases like PostgreSQL work best with individual columns, not arrays. Separate columns are:
- More queryable (you can `WHERE optionA = 'something'`)
- Safer (the DB enforces that all four always exist)
- More standard

The service layer (`questionService.ts`) handles the conversion in both directions:
- **DB → frontend:** `[optionA, optionB, optionC, optionD]` → `options` array (`toTQuestion`)
- **Frontend → DB:** `options[0..3]` → `optionA..optionD` (`fromTQuestion`)

### The app factory pattern

`app.ts` exports a `createApp()` function instead of a single `app` object:

```ts
// app.ts
export const createApp = () => {
  const app = express();
  // ... setup
  return app;
};

// index.ts
const app = createApp();
app.listen(config.port);
```

**Why?** Testability. In tests you can call `createApp()` to get a fresh Express instance without starting a real server on a port. Supertest handles the actual HTTP binding internally. If `app.ts` called `app.listen()` itself, importing it in tests would start a real server every time.

### Layered architecture

The server code is split into three layers, each with one job:

```
Routes → Service → Database
```

| Layer | File | Responsibility |
|---|---|---|
| **Routes** | `routes/questionRoutes.ts` | Parse HTTP requests, call service, send HTTP responses |
| **Service** | `services/questionService.ts` | Business logic, Prisma queries, data mapping |
| **DB** | PostgreSQL via Prisma | Persist data |

**Why separate routes from services?**

Routes deal with HTTP (status codes, `req`, `res`). Services deal with data. Keeping them separate means:
- You can test the service without HTTP
- You can swap the HTTP framework without touching business logic
- Each file has a clear, single responsibility

### Middleware

Middleware is a function Express calls between receiving a request and sending a response. The signature is always:

```ts
(req: Request, res: Response, next: NextFunction) => void
```

Calling `next()` passes control to the next middleware. Calling `next(error)` skips to the error handler.

**`validateRequest.ts`** — runs before route handlers that need body validation:
```
Request → validateRequest (checks Zod schema) → route handler → Response
                          ↓ on failure
                      errorHandler (sends 400)
```

**`errorHandler.ts`** — a special 4-argument middleware Express uses for errors:
```ts
(error, req, res, next) => void   // 4 args = error handler
```
It handles three cases:
- `ZodError` → 400 (bad input)
- Prisma "record not found" (`P2025`) → 404
- Everything else → 500

### The Vite proxy

In development, the frontend (port 5173) and backend (port 3001) run on different ports. Normally this causes **CORS errors** because the browser blocks requests from one origin to another.

The Vite proxy solves this by forwarding any `/api/*` request from the frontend to the backend, so from the browser's perspective everything comes from the same origin:

```
Browser → Vite dev server (5173) → Express server (3001)
         /api/questions              /api/questions
```

In production you'd configure a real reverse proxy (e.g. Nginx) to do the same thing.

### The static fallback

`questionsApi.ts` wraps each fetch in a `try/catch`:

```ts
export const fetchAllQuestions = async (): Promise<TQuestion[]> => {
  try {
    const res = await fetch("/api/questions");
    if (!res.ok) throw new Error();
    return (await res.json()).data;
  } catch {
    console.warn("API unavailable, falling back to static data");
    return staticQuestions;  // imported from src/data/questions.ts
  }
};
```

This means the app degrades gracefully: if the backend is down, it still works using the bundled static data. This pattern is called **graceful degradation**.

---

### Why `dayHelpers` now takes a `questions` parameter

Before:
```ts
// imported static data at the top of the file
import { questions } from "../data/questions";

export const getQuestionsForDay = (day: number): TQuestion[] => {
  return questions.slice(...);
};
```

After:
```ts
// no static import — caller provides the data
export const getQuestionsForDay = (day: number, questions: TQuestion[]): TQuestion[] => {
  return questions.slice(...);
};
```

**Why?** The data now comes from an async API call, not a synchronous import. The route component fetches the questions and then passes them into the helper. This is called **dependency injection** — instead of a function reaching out and grabbing its own data, the caller provides the data as an argument. It also makes the function easier to test (just pass in any array you want).

## The full request lifecycle

When the frontend loads `QuizRoute`:

```
1. Component mounts
2. useEffect calls fetchAllQuestions()
3. fetch("/api/questions") → Vite proxy → Express
4. questionRouter GET "/" → getAllQuestions() → Prisma → PostgreSQL
5. Rows come back → toTQuestion() maps columns to options array
6. Express responds with { data: TQuestion[] }
7. Component receives data, setAllQuestions(), setLoading(false)
8. buildQuizSet(day, mode, allQuestions) creates the quiz
9. Component renders
```

If step 3–6 fails (server down), step 6 never resolves and the catch block returns the static array instead.
